## 🛠️ **The 16 task optimization strategies**

This lesson covers the **three strategy categories** and the **specific strategies within each**. Use the introductory table for each category as a quick reference, then read the per-strategy detail when applying one.

---

## 🎯 **Trigger Strategies**

These regulate **how automations are triggered** — controlling the **number of jobs** for a given recipe (factor #1 from 7.1).

|Strategy|One-line summary|
|---|---|
|**⏰ Polling Frequency**|Set polling interval to a higher value.|
|**🔍 Trigger Conditions**|Filter only the events the recipe should process.|
|**📦 Batch/Bulk Triggers**|Process multiple events in a single job.|

---

### ⏰ Polling frequency

**The strategy:** set the polling interval to a higher value. Triggers also support custom intervals.

Most Workato triggers are poll-based, defaulting to **5-minute intervals**. Lower polling intervals create more jobs and higher task consumption — especially when pre-processing logic (like data enrichment) sits on top.

> 📌 **Use when** the source system doesn't have frequent new/updated records, AND/OR the use case isn't time-critical.

---

### 🔍 Trigger conditions

**The strategy:** use trigger conditions to filter only the events that the recipe should process.

Trigger conditions are additional rules that determine which events get selected. _Example:_ process only "closed-won" opportunities.

> 📌 **No task is consumed if the conditions are not met** — trigger conditions eliminate extraneous jobs entirely.

> 📌 **Use when** you only need to process records that fulfill specific event-data criteria.

---

### 📦 Batch/bulk triggers

**The strategy:** use batch triggers when available to process multiple events in a single job.

Batch triggers can process **hundreds to thousands** of records together. Larger batch sizes cut the number of jobs needed for the same volume of records — and therefore the task count.

> ✅ **Best practice:** combine batch triggers with **batch actions** in the same recipe for substantial gains.

> 📌 **Use when** you have data synchronization use cases with high volumes — and the source system supports batch/bulk actions.

---

### 🧠 Quick recall — trigger strategies

- Default Workato polling interval is `_____` minutes. (5)
- The strategy that "eliminates" jobs entirely when conditions aren't met? (Trigger conditions — no task consumed if condition fails.)
- Best practice when using batch triggers? (Combine with batch actions in the recipe.)
- Name the three trigger strategies. (Polling frequency, trigger conditions, batch/bulk triggers)

## 🏎️ **Batch Strategies**

These optimize **multiple events processing in a single job** (factor #2 from 7.1).

|Strategy|One-line summary|
|---|---|
|**📦 Batch/Bulk Actions**|Process multiple events in a single step.|
|**🔎 Filtering Events**|Apply WHERE / query params in search-style actions to limit returns.|
|**🔄 Delta Sync**|Process only changed records, not the full set.|
|**📥 ELT Pipeline**|Use ELT instead of ETL for complex transformations on data-warehouse platforms.|
|**🌊 Streaming**|Use streaming actions for large file transfers.|

---

### 📦 Batch/bulk actions

**The strategy:** use batch/bulk actions when available to process multiple events in a single step.

Batch actions can process hundreds to thousands of records together. With custom batch size:

> 📌 Instead of processing 1 record (1 task) at a time, you process **X-record batches (still 1 task)**. Task usage drops by a factor of X.

> 📌 **Use when** doing data sync with high volumes — and the target system supports batch/bulk actions.

---

### 🔎 Filtering events

**The strategy:** use action-specific filtering criteria — WHERE conditions, query parameters in APIs — to limit the events returned from search/list actions.

Recipes often only need a subset of data based on business logic or dynamic request data. Applying filters in the **search or list action itself** limits results from the source application directly — eliminating extra filtering steps in the recipe.

> 📌 **Use when** you need to filter a batch of events or a list input from search/get actions before using them in the recipe.

---

### 🔄 Delta sync

**The strategy:** process only changed records instead of a full dataset.

Delta synchronization processes only data that has **changed** (new or updated) since the last successful job execution. Use identifiers in source data — like a "last modified date" field — to read only the changed records.

Benefits: improved data consistency, efficient task consumption, reduced load on source and target systems.

> 📌 **Use when** doing data sync where the source system supports change identification.

---

### 📥 ELT pipeline

**The strategy:** use the ELT pattern instead of ETL for complex data transformations.

**ELT** (Extract → Load → Transform) loads data into the data warehouse first, then transforms it there. This takes advantage of purpose-built warehouses (e.g. **Snowflake**) for high-volume processing while Workato handles orchestration.

> 📌 **Use when** you have high-volume data processing requirements with a modern data warehouse.

---

### 🌊 Streaming

**The strategy:** use streaming actions to automate large file transfers.

**File streaming** reads and writes files in smaller chunks in sequence. _Example:_ transferring records from SFTP to Amazon S3 — Workato splits the file into chunks, downloads them, then uploads each in a separate request.

> 📌 **Use when** doing file management or analytics where both source and target systems support streaming.

---

### 🧠 Quick recall — batch strategies

- Batch actions reduce task usage by what factor compared to row-by-row? (Factor of X, where X is the batch size.)
- Process only changed records since the last successful job — what's this called? (Delta sync)
- For high-volume warehousing with Snowflake-style targets, which pattern fits? (ELT)
- Which strategy splits a large file into chunks for transfer? (Streaming)
- Filtering with WHERE / query params at the action level avoids what? (Additional filtering steps in the recipe.)

## ⚡ **Efficiency Strategies**

These use **alternative approaches to implementing business logic** more efficiently (factor #3 from 7.1).

|Strategy|One-line summary|
|---|---|
|**🧮 Formula Mode**|Inline formulas instead of repeat actions with lists.|
|**📋 List Processing**|Dynamic list mode for implicit looping.|
|**💎 Collections**|Use Collections by Workato for in-memory SQL on data from multiple sources.|
|**💻 Custom Code**|Ruby/JavaScript/Python for compute-intensive processing.|
|**🎁 Variable Grouping**|Declare multiple variables in a single step.|
|**📍 Variable Declarations**|Declare variables close to where they're used (avoid unused-variable tasks).|
|**💾 Response Caching**|Cache API GET responses for frequently-accessed data.|
|**🛠️ Custom SDK**|Connector SDK for complex automation logic in Ruby.|

---

### 🧮 Formula mode

**The strategy:** use inline formulas for data transformations instead of repeat actions with lists.

Business apps accept and return complex data structures (JSON objects, arrays). Formulas — whitelisted Ruby methods — let you work with and format data easily.

> 📌 **Mapping data in formula mode handles complex data structures and counts as just 1 task per step**, vs. potentially many tasks with repeat actions.

> 📌 **Use when** the recipe needs validation, cleansing, extraction, type casting, or transformations.

---

### 📋 List processing

**The strategy:** define a target list schema and use dynamic list mode for **implicit looping** of existing lists.

Workato can automatically loop over an input list and apply configured datapill transformation logic to each element. To use this:

- **Mapper by Workato** — define the target list schema using Common Data Models from the tools menu.
- **Accumulate List** action — edit the schema directly in the recipe to define list item fields as an array.

> 📌 **Use when** you need to apply datapill-level transformation to every element in an existing list before using it elsewhere.

---

### 💎 Collections

**The strategy:** use **Collections by Workato** for complex data processing.

Collections by Workato uses an **in-memory database**. You can create lists and query them using **standard SQL syntax** with `WHERE`, `GROUP BY`, `JOIN`, etc., to manipulate data into your desired format.

> 📌 **Use when** doing complex processing — merging/enriching data from different sources, applying transformations to a batch of events, aggregating data for summary views.

---

### 💻 Custom code

**The strategy:** use custom code for complex business logic and data processing.

Compute-intensive processing — flattening hierarchical data, filtering lists, transforming object maps, nested loops — can be done much more efficiently with a few lines of code.

> 📌 Workato supports **Ruby, JavaScript, and Python**. **Each execution of custom code counts as 1 task**.

> 📌 **Use when** doing hierarchical data, high/nested cardinality, custom parsing, semi-structured or unstructured data processing, list filtering.

---

### 🎁 Variable grouping

**The strategy:** use a single **Variables by Workato** step to create multiple variables.

Creating each variable as its own step adds up quickly in high-frequency recipes. Since **Spring 2021**, Variables by Workato supports declaring multiple variables in a single step — counted as a single task.

> 📌 **Use when** optimizing older recipes (pre-Spring-2021) or building new ones — group related variable declarations into one step.

---

### 📍 Variable declarations

**The strategy:** declare and define variables **close to where they're first used**.

A common anti-pattern: declaring multiple variables upfront for use inside a conditional block that may never execute. Each unused declaration still counts as a task.

> 📌 **Use when** you have variables intended for use inside a conditional block — declare them inside the block, not before it.

---

### 💾 Response caching

**The strategy:** use cache response for API **GET** request endpoints.

Workato can cache API responses for GET endpoints with a configurable lifetime — **up to 3,600 seconds (1 hour)**. For any subsequent request matching the cache key, Workato serves the cached response.

> 📌 **Cached responses bypass the API endpoint recipe entirely and count as 1 task** — large savings on high-volume read endpoints.

> 📌 **Use when** implementing an API endpoint for frequently-accessed data or multiple concurrent requesters sharing the same data.

---

### 🛠️ Custom SDK

**The strategy:** use the Workato Connector SDK to implement complex automation logic.

The Custom SDK connector lets you write business logic in **Ruby** while exposing a consistent low-code interface for recipes. The SDK also supports advanced capabilities that optimize task count.

> 💡 _Example:_ a multi-step action to get asynchronous, long-running Google BigQuery results — implemented via SDK — can be counted as just **1 task**.

> 📌 **Use when** implementing complex automation logic (asynchronous polling) or leveraging Ruby methods unique to the connector SDK.

---

### 🧠 Quick recall — efficiency strategies

- Inline formula vs repeat action — task count? (Formula = 1 task per step regardless of complexity; repeat actions multiply tasks per iteration.)
- Three custom code languages supported in Workato? (Ruby, JavaScript, Python)
- Each custom code execution counts as `_____` task(s). (1)
- API response cache max lifetime? (`_____` seconds) (3,600 — 1 hour)
- Why does declaring variables inside (not before) a conditional block save tasks? (If the block never executes, the variable is never created — saving the declaration's task cost.)
- Spring 2021 feature in Variables by Workato? (Declaring multiple variables in a single step — counted as one task.)
- Workato Collections uses what kind of database? (In-memory)

---

## 🚀 **Module key takeaways**

- **3 trigger strategies** control how often jobs fire: longer polling, trigger conditions, batch triggers.
- **5 batch strategies** pack more events into single jobs/actions: batch actions, action-level filtering, delta sync, ELT pipelines, streaming.
- **8 efficiency strategies** shrink the task count per logical operation: formula mode, list processing, Collections, custom code, variable grouping, variable declarations, response caching, custom SDK.
- **Recurring 1-task-per-step pattern**: formula mode, custom code, cached responses, batch actions, grouped variables — all count as **1 task each** regardless of underlying complexity. This is the core insight powering most optimization wins.
