## ⭐ **Best practices for integrating databases**

Using Workato's database tools effectively means **reducing database actions to minimize load and strain**, and **splitting complex workflows into separate recipes** to keep your Workato assets manageable. Three areas matter most: recipe step design, combining commands before calling the database, and how the database itself is structured.

---

### 🎯 Choose efficient recipe step designs

Pick the right action shape so you don't make extra round-trips to the database:

- **🔄 Use `Upsert` actions** instead of separately calling Select, then Update or Insert. One action with built-in logic beats two with manual branching.
- **📚 Use Lists by Workato and Batch actions** to aggregate data and send it to the database in fewer calls instead of one row at a time.

> 📌 The general principle: **fewer database actions = less load and faster recipes.** Every round-trip to the database costs time and tasks.

---

### 🧮 Combine commands before calling a database

Push transformation work _into_ the database (or into Workato's SQL utilities) rather than running many small actions:

- **🧩 Use stored procedures, custom SQL functions, or SQL Collections** to transform data in one step and return only the necessary columns before sending back to Workato.
- **♻️ Use Recipe functions and API recipes** to reduce redundancy in recipe steps by creating reusable, callable recipes.

---

### 🏗️ Structure the database format well

The integration itself can only be as clean as the underlying schema:

- **🔑 Define a primary key** — a column that uniquely identifies each row, like an auto-incrementing number. Workato actions rely on this to locate and update rows reliably.
- **✅ Add constraints** to the columns of tables (via `CREATE` or `ALTER`) to validate data flowing into the database — bad data is much cheaper to reject at the schema level than to clean up downstream.

---

### 🛠️ Further database applications

A few common patterns worth keeping in mind:

- **➕ Add more actions** in the workflow when more complex data integration is needed.
- **🔍 Filter rows at the trigger** by using **"New/Updated rows via Custom SQL"** or **"Scheduled query"** actions in SQL Server (and other app connectors). Filtering early is cheaper than processing then discarding.
- **♻️ Create Recipe functions with parameters** that you can input from the parent recipe when mapping datapills — useful for moving data between databases with a reusable pattern.

---

### 🧠 Quick recall

- Recommended over running a Select followed by Update or Insert? (`_____`) (Upsert)
- Two Workato tools for aggregating data before sending to a database? (Lists by Workato; Batch actions)
- What's the single best schema decision for reliable Workato integration? (A well-defined primary key — ideally auto-incrementing.)
- Where should data validation happen — in Workato or at the database schema? (Both, but database-level constraints are the cheapest way to reject bad data.)
- Which trigger-side action lets you filter rows with custom SQL? (New/Updated rows via Custom SQL — also Scheduled query for time-based filtering.)
- What's the pattern for making one recipe callable from another with parameters? (Create a Recipe function with input parameters.)

---

## 🚀 **Module key takeaways**

- **Fewer database actions = better recipes** — Upsert beats Select-then-Update; batched lists beat row-by-row.
- **Push work to the database** when possible (stored procedures, custom SQL, SQL Collections) — return only what you need.
- **Recipe functions and API recipes** are the reusability tools for avoiding redundant steps across multiple recipes.
- **A solid primary key** (e.g. auto-incrementing) and **schema-level constraints** make integration significantly easier downstream.
- **Filter early** at the trigger level (`New/Updated rows via Custom SQL`, `Scheduled query`) rather than pulling everything and discarding.
