## 💻 **SQL commands**

SQL commands are used to call and transform data in databases. Each database connector has its own set of applicable SQL commands. While many Workato database connectors have **built-in actions** for simple manipulations (insert, update, delete rows), you can also **combine SQL commands** for more specific changes.

> 📌 Two categories of SQL commands are most relevant in controlling, maintaining, and optimizing data: **DDL** and **DML**.

|Category|Stands for|Purpose|
|---|---|---|
|**🛠️ DDL**|Data **Definition** Language|Defines and maintains database structure (tables, indexes, views).|
|**✏️ DML**|Data **Manipulation** Language|Manipulates the data inside the database (rows).|

A simple memory hook: **DDL changes the schema; DML changes the data**.

---

### ✏️ DML — Data Manipulation Language

DML commands act on rows of data.

|Command|What it does|Example|
|---|---|---|
|**`INSERT`**|Add rows to a table.|`INSERT INTO example_table VALUES (35, 'Aden', 'Workato');`|
|**`UPDATE`**|Locate existing rows and change their values.|`UPDATE customers SET email = 'example@gmail.com' WHERE customer_id = 58;`|
|**`DELETE`**|Remove specified rows.|`DELETE FROM storage_items WHERE type = 'badges' OR quantity = 0;`|

---

### 🛠️ DDL — Data Definition Language

DDL commands act on database objects (tables, views, indexes).

|Command|What it does|Example|
|---|---|---|
|**`CREATE`**|Add a new object to a database (table, view, index).|`CREATE TABLE example_table (name VARCHAR(50), ... birthday DATE);`|
|**`ALTER`**|Change the structure of an existing object — add/change/delete columns, rename objects, modify indexes, procedures, or triggers.|`ALTER TABLE example_table ADD age INT;`|
|**`DROP`**|Remove an existing object from the database.|`DROP TABLE example_table;`|

---

### 📝 Running custom SQL commands in actions

Beyond built-in actions, Workato lets you write custom SQL inside database connector actions. This improves recipe efficiency and balances load between Workato and the database.

|Action|When to use it|Key detail|
|---|---|---|
|**Run Custom SQL**|General custom SQL — send a SQL string to the connector instance to be executed.|Most common path; improves efficiency by pushing work to the database.|
|**Run long query using custom SQL**|Queries on **large datasets** where response time could exceed the **60-minute timeout** and cause recipe failure.|**Asynchronous** execution — bypasses the timeout.|
|**Select rows using custom SQL**|Reading rows based on a custom query.|Returns **100 rows by default**.|

> 📌 **The 60-minute rule:** if a query might run longer than 60 minutes, use **"Run long query using custom SQL"** to execute it asynchronously. Otherwise the recipe will time out and fail.

- DDL changes the `_____`; DML changes the `_____`. (Schema/structure; data/rows)
- Name the three DML commands. (INSERT, UPDATE, DELETE)
- Name the three DDL commands. (CREATE, ALTER, DROP)
- Which action handles queries that may exceed the 60-minute timeout? (Run long query using custom SQL — asynchronous)
- How many rows does "Select rows using custom SQL" return by default? (`_____`) (100)
- Which command would you use to add a new column to an existing table? (ALTER)

## 🧩 **Stored procedures**

> 📌 **Stored procedures** are collections of SQL statements **defined in a database** that can be executed as a predefined set of operations by calling the procedure name. They're a **reusable** block of code accessible and runnable from different applications or queries.

---

### 🎯 The role of stored procedures

Stored procedures serve two main roles in Workato workflows:

1. **🎼 Orchestration of complex operations** — execute complex transactions or loads with Workato as the orchestrator that calls the stored procedure at the right time.
2. **🔧 Versatile tasks with input parameters** — perform varied operations using input parameters, while maintaining control over the scope of recipes operating on the database.

---

### ▶️ Using stored procedures in actions

Some Workato database connectors include an **"Execute stored procedure"** action. This action lets you:

- Choose a stored procedure by name.
- Select parameter inputs to pass into it.
- Execute it against the connected database.

This is how Workato hooks into stored procedures that already exist in your database — Workato doesn't define the procedure, your DBA does.

---

### 🧠 Quick recall

- Stored procedures are defined **in the database** — true or false? (True)
- Can stored procedures receive input parameters? (Yes — that's one of their two main roles.)
- The Workato action used to call a stored procedure is named `_____`. (Execute stored procedure)
- Name the two roles stored procedures play in Workato workflows. (Orchestrating complex operations; performing versatile tasks via input parameters)

---

## 🚀 **Module key takeaways**

- **SQL commands** split into two categories: **DML** (data — INSERT, UPDATE, DELETE) and **DDL** (definition — CREATE, ALTER, DROP).
- Workato exposes three **custom SQL actions**: Run Custom SQL (default), **Run long query** (async, for queries that might exceed 60 minutes), and Select rows (100 by default).
- **Stored procedures** are reusable SQL blocks _defined in the database itself_. Workato calls them via the **"Execute stored procedure"** action.
- Stored procedures shine for **complex transactions** and **parameterized reusable tasks** — Workato handles orchestration, the database handles the heavy SQL.
