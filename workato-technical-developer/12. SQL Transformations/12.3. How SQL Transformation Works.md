## ⚖️ **SQL Transformations vs SQL Collections**

Both tools sit under the umbrella of "SQL in Workato" — but they're built for **different data profiles**. Choosing correctly matters.

> 📌 Side-by-side comparison:

| |**🔧 SQL Transformations**|**💎 SQL Collections**|
|---|---|---|
|**Data volume**|Ideal for **large-volume data**|Limited to smaller datasets|
|**Bulk data**|Supports **bulk data transformation**|Cannot support bulk data|
|**Output reuse**|Output data can be **reused across multiple jobs/recipes**|Data is **confined within a specific job**|
|**Recipe design**|Streamlined — combines source setup + query + output config into a **single action**|Multiple actions across create/insert/query steps (chapter 6)|

**The decision rule:**

- **📈 Large data, cross-job persistence needed** → SQL Transformations
- **⚡ Small data, single-job scope** → SQL Collection (chapter 6)

Refer back to **chapter 6.2's comparison table** for the full context on where SQL Collection fits alongside DataTables, FileStorage, and Lookup tables.

## 🧱 **The three components of SQL Transformations**

Every SQL Transformation is composed of three moving parts working together.

---

### 📦 Component 1: Bulk extraction

- **📈 Efficient way to retrieve data** in one operation.
- **🚀 Useful for handling massive datasets**.
- **⚡ Enables faster retrieval** than row-by-row approaches.

> 📌 **Bulk extraction is a required component** of SQL Transformations — it's not an optional optimization, it's how the tool fundamentally pulls data.

---

### 📄 Component 2: CSV files

- **📊 Provide a structured format** for the data.
- **✏️ Easy to transfer, manipulate, and transform**.
- **🚚 The medium for storing and moving data** before applying the SQL transformation.

CSV is the interchange format that lets SQL Transformations normalize data from many sources into one queryable shape.

---

### 🔧 Component 3: SQL Transformations (the query engine)

- **🔗 Apply SQL queries** to filter, aggregate, and join.
- **🧮 Process data with powerful functions**.

This is where the actual SQL work happens — after bulk extraction has staged the data as CSV.

---

## 📐 **Schema handling — two approaches**

When SQL Transformations processes CSV data, the **column types have to come from somewhere**. Two options:

---

### 🔍 Inferred schema

- **🤖 Detects structure based on CSV file headers.**
- **🔤 Parses data and assigns the most suitable data types automatically.**
- **⚙️ Not manual** — no configuration required.

**When to use:** working with **loosely structured or changing data** — where the schema might evolve or vary between runs.

---

### 🎯 Defined schema

- **✍️ Manually define a schema** for consistency and control.
- **🎯 Ideal when you need precision** — specific types, strict column ordering, guaranteed structure.

**When to use:** production pipelines where **type precision matters**, or when data quality gates depend on strict schema.

> 📌 **Decision rule:** inferred = flexibility, defined = precision. If schema changes or is uncertain → inferred. If schema is stable and precision matters → defined.

---

### 🧠 Quick recall

- Which tool is right for **large-volume, cross-job** work? (SQL Transformations)
- Which tool is right for **small, single-job in-memory** work? (SQL Collection)
- Name the three components of SQL Transformations. (Bulk extraction; CSV files; SQL Transformations query engine)
- What role do CSV files play in SQL Transformations? (The interchange format — for storing and moving data before applying the transformation.)
- Name the two schema-handling approaches. (Inferred; defined)
- Which schema approach is better for changing data? (Inferred — auto-detects from CSV headers.)
- Which schema approach is better for precision? (Defined — manually specified.)
- One key architectural advantage SQL Transformations has over SQL Collections in recipe design? (Combines source setup + query setup + output configuration into a **single action**.)

## 🚀 **Module key takeaways**

- **SQL Transformations ≠ SQL Collections** — different volume/persistence/scope profiles. Learn both, choose per use case.
- **Three components**: bulk extraction (pull), CSV files (stage), SQL transformations (query).
- **Two schema approaches**: **inferred** (auto from headers, best for changing data) vs **defined** (manual, best for precision).
- **Single-action recipe design** — one of SQL Transformations' key ergonomic wins vs. multi-step SQL Collections.
