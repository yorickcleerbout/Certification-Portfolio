## ⭐ **Best practices**

Two foundational practices for effective SQL Transformations:

- **📦 Use bulk extraction + CSV format** — extract data in bulk and use CSV for structured data. This is the efficient-processing baseline.
- **🧹 Validate and cleanse data** — before starting transformation, remove duplicates and errors, ensure data quality. Skipping this leads to bad output that's hard to trace back.

---

## ⚠️ **Common pitfalls & how to avoid them**

### ❌ Pitfall 1: Data quality issues

Data containing **errors, duplicates, or missing values** leads to incomplete or inaccurate results.

> ✅ **How to avoid:** always **validate and clean your data before starting the transformation** — this ensures the transformation yields correct output.

---

### 🐌 Pitfall 2: Performance bottlenecks

Inefficient SQL queries can slow down the transformation process.

> ✅ **How to avoid:** always **use bulk actions for efficiency** — never process one record at a time when a batch will do.

---

## 🚀 **Optimization strategies**

Two ways to make SQL Transformations perform better:

---

### 📁 Optimization 1: Leverage Workato FileStorage

- **📁 Store data in Workato FileStorage**.
- **🔁 Retrieve and process files across jobs** — persistence unlocks reuse.

FileStorage is the strategic staging layer for large datasets — using it well is one of the biggest performance levers.

---

### 📐 Optimization 2: Schema handling

Pick your schema strategy based on the data profile:

- **🔍 Inferred schemas** — useful when working with **loosely structured or changing data**.
- **🎯 Defined schemas** — ideal when you need **precision, handling, and structure**.

> 📌 The choice is a knob, not a rule. Inferred is easier and more flexible; defined is stricter and more reliable. See **12.3** for the decision framework.

---

## 🍳 **Recipe best practices**

Two practices for how SQL Transformations should live inside recipes:

- **📝 Define SQL queries clearly** — use SQL queries to explicitly manipulate the data. Named, purposeful queries beat ad-hoc mystery queries.
- **📤 Store transformed data as CSV in Workato FileStorage** — persistent output that can be consumed by downstream jobs and recipes.

---

- Name the two foundational best practices. (Bulk extraction + CSV; validate and cleanse data)
- What's the pitfall of skipping data validation? (Errors, duplicates, and missing values lead to incorrect or incomplete output.)
- How do you avoid performance bottlenecks? (Use bulk actions for efficiency — never row-by-row when batches will work.)
- Where should you store data for cross-job reuse? (Workato FileStorage)
- When is an inferred schema the right choice? (Loosely structured or changing data — where flexibility matters more than precision.)
- When is a defined schema the right choice? (When you need precision, strict handling, and consistent structure.)
- What's the recommended output storage strategy for SQL Transformations? (Store transformed data as CSV files in Workato FileStorage.)

---

## 🚀 **Module key takeaways**

- **Bulk + CSV + FileStorage** — the three magic ingredients for good SQL Transformations performance.
- **Always validate and clean before transforming** — bad input = bad output, and bad output is very hard to trace back to the source data issue.
- **Use bulk actions**, not per-row loops — for both correctness and speed.
- **Schema choice = flexibility vs precision**: inferred for changing/loose data, defined for precision.
- **FileStorage everywhere** — as staging area for large inputs, and as the persistent storage for transformed outputs. Reused across jobs.
