## 💼 **Why do we need OPA?**

Enterprises have applications and databases — on-prem apps deployed inside their corporate datacenter — that are **deliberately not accessible to cloud services like Workato**, for security reasons.

> 📌 OPA is required so Workato users can build recipes that **communicate with on-prem applications securely**. The agent provides selective, controlled access — you decide which on-prem apps Workato recipes can reach and at what level.

The configuration is intentional and granular: instead of opening blanket access, you authorize specific on-prem apps for specific recipes.

---

## 🚀 **Use case: Employee onboarding**

A typical company hires tens or hundreds of employees per year. With manual onboarding processes — accounts, devices, access provisioning — coordinating it all becomes a major burden.

### 🤖 The automated flow

Using OPA, the entire onboarding process can be streamlined end-to-end. The flow:

1. **🎬 Trigger** — a new employee is hired in **Workday** (the applicant tracking system).
2. **💻 IT provisioning** — the onboarding workflow automatically requests IT assets (work laptop, work email) via **ServiceNow**, the IT service app.
3. **🔐 On-prem identity write** — employee information and assigned devices are written to **Active Directory** — the company's on-prem identity provider, which is the **single source of truth for employee access privileges**.
4. **📣 Notification** — once the on-prem record is written, the requester is notified automatically via **Slack** that the employee has been provisioned.

> 📌 The critical step is #3: writing to **Active Directory**, which lives on-prem and is unreachable from the cloud without OPA. Without OPA, this workflow can't be fully automated.

---

### 🧠 Quick recall

- Why do enterprises need OPA? (On-prem apps inside the corporate datacenter aren't accessible to cloud services like Workato for security reasons — OPA bridges that gap securely.)
- Does OPA give Workato blanket access to every on-prem app? (No — access is **selective and controlled** through configuration.)
- In the onboarding example, what triggers the workflow? (A new hire in Workday — the applicant tracking system.)
- Which on-prem app in the example holds the single source of truth for employee access? (Active Directory)
- Which two cloud apps participate in the example flow alongside Workday and Active Directory? (ServiceNow for IT provisioning; Slack for notifications.)

---

## 🚀 **Module key takeaways**

- **OPA exists because** enterprise on-prem apps are deliberately walled off from cloud services. OPA provides a controlled bridge.
- **Selective, configurable access** — you decide which on-prem apps each recipe can reach and at what level.
- **Classic use case**: automated onboarding flows that span cloud (Workday, ServiceNow, Slack) and on-prem (Active Directory) systems — impossible without OPA bridging the on-prem write step.
