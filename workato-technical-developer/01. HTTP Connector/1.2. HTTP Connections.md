## 📡 **Brief API overview**

> 📌 **API** stands for **Application Programming Interface** — a mechanism that allows two applications to exchange information using a set of rules.

When end users interact with an application, they do it through a user interface — viewing data and performing actions. An API plays the same role, just between _applications_: it provides a way for one application to interact with another.

A useful analogy: APIs are like waiters at a restaurant.

- The **menu** is the **API documentation** (or **WSDL** — Web Services Description Language). It lists what the application offers.
- You **order** what you want by making an **API call** from within another application.
- The API delivers your order to a database and **returns with the data you requested**.

---

## 🌐 **Web services**

If an application has an API, you can use the HTTP Connector or Connector SDK to connect through **web services**. There are two common types of web service APIs: **SOAP** and **REST**.

---

### ⚖️ SOAP vs REST

|Characteristic|REST|SOAP|
|---|---|---|
|**Protocol**|Architectural style protocol|XML-based message protocol|
|**Communication**|XML or JSON to send and receive data|WSDL between consumer and provider|
|**Calls**|Basic call services via URL path|Invokes services by calling RPC methods|
|**Returns**|Readable as plain XML or JSON|Not human-readable|
|**Transport**|HTTP only|HTTP, SMTP, FTP, and other protocols|
|**JavaScript**|Easy to call from JavaScript|Difficult to implement|
|**Performance**|Less CPU-intensive, lean code|Heavier than REST|

> 📌 Although REST supports both XML and JSON, **most REST services use JSON**. **SOAP only uses XML**.

SOAP also uses WSDL — through which a provider describes the functionality offered by a web service to the consumer, abstractly defining operations and messages that are then bound to a concrete network protocol and endpoint. In REST, you get endpoint information from the API documentation provided by the application.

---

### 📜 XML

**XML (Extensible Markup Language)** is a markup language that defines rules for encoding documents in a format that's both human- and machine-readable. The same family of languages includes HTML, used to format web pages.

```xml
<?xml version="1.0"?>
<List>
   <Student id="101">
      <Name>John</Name>
      <Place>London</Place>
   </Student>
   <Student id="102">
      <Name>Peter</Name>
      <Place>Sydney</Place>
   </Student>
   <Student id="103">
      <Name>Mary</Name>
      <Place>Frankfurt</Place>
   </Student>
</List>
```

---

### 💎 JSON

**JSON (JavaScript Object Notation)** is an open standard file format that uses **human-readable text** to store and transmit data objects consisting of attribute–value pairs and arrays.

```json
{
   "Students": [
      { "id": "101", "Name": "John", "Place": "London" },
      { "id": "102", "Name": "Peter", "Place": "Sydney" },
      { "id": "103", "Name": "Mary", "Place": "Frankfurt" }
   ]
}
```

> 📌 JSON is more **compact and lighter** than XML (better performance), and **easier to read**.

The Workato HTTP Connector supports **JSON and XML on REST**, as well as **URL-encoded forms**.

## 📞 **How Workato HTTP Connector communicates with APIs**

Communication with any API via the HTTP Connector has three aspects:

1. **🔐 Authorization** — credentials such as username/password, bearer token, etc.
2. **📤 Request** — a request message sent to the HTTP server.
3. **📥 Response** — a status code and response message, typically containing the requested resource.

---

### 🔐 Secure connections

A secure connection means **authentication + authorization**. To communicate with APIs, you must establish a connection using a supported scheme. The Workato HTTP Connector supports:

- None
- Basic
- Header auth
- Query params
- Custom
- OAuth 2 (authorization code grant)
- OAuth 2 (client credentials grant)
- AWS access key auth
- AWS IAM role auth

For details on each, see the [Workato HTTP Connector documentation](https://docs.workato.com/developing-connectors/http-v2.html#http-connector-v2).

> 📌 Two ways Workato communicates with APIs:
> 
> - **HTTP action** — when an API endpoint exists but no OOB action covers it.
> - **HTTP Connector** — when an API endpoint exists but no OOB connector covers the application.

---

### 📨 Response and status codes

Workato reads status codes and maps code and data responses to recipe datapills as part of action output.

> 📌 A status code is a **three-digit integer**. The first digit identifies the category:

|Code range|Meaning|
|---|---|
|**1xx**|Informational message|
|**2xx**|Success|
|**3xx**|Redirect to another URL|
|**4xx**|Client-side error|
|**5xx**|Server-side error|

---

- A 503 status code means? (Server-side error — 5xx category)
- Which two auth grants does OAuth 2 support in the HTTP Connector? (Authorization code grant, client credentials grant)

## 🌐 **Base URLs**

> 📌 A **Base URL** is the core, consistent component of a web address that stays the same across pages — typically the protocol, domain name, and main directory. It acts as the starting point from which relative paths derive absolute URLs.

In the HTTP Connector, Base URLs are used to:

- **🚧 Restrict outbound requests** — anyone using the connection can only connect to the server or application you specify.
- **🛡️ Secure your data** — sensitive data only passes through a specified endpoint, domain, or host.
- **🔢 Invoke the correct API version** — once configured, all requests route to the same endpoint including the API version. This also makes API version migrations easier (update the Base URL, all recipes follow).

---

### 🧮 How it works

When an HTTP action executes, the Base URL and the action's Request URL combine to form the final request URL:

> 📌 **Base URL + Action Request URL = Final Request URL**

Example:

- Base URL: `https://instance.salesforce.com`
- Action Request URL: `/services/data`
- Final request URL: `https://instance.salesforce.com/services/data`

---

### ✅ URL validation rules

Workato restricts all outgoing HTTP requests to match the configured Base URL. There are three valid patterns:

|Pattern|Behavior|
|---|---|
|**Absolute path**|Base URL `https://instance.salesforce.com` + Request URL `https://instance.salesforce.com/services/data/` — Workato validates that the base of the given endpoint matches the connection's Base URL.|
|**Relative path with `/` prefix**|Base URL `https://instance.salesforce.com/service` + Request URL `/services/data/` — Workato validates the path starting from the root domain (`.com`). `/service` in the Base URL must match the start of the Request URL.|
|**Relative path without `/` prefix**|Base URL `https://instance.salesforce.com/service` + Request URL `data/` — Workato **appends** the Request URL to the Base URL.|

> ⚠️ **Relative paths can only be used if a Base URL is configured.** Without a Base URL, only absolute paths work.

If the Request URL attempts to use a different Base URL — or if the computed final URL isn't valid — Workato **prevents the HTTP action and stops the job with an error**.

Common Base URL errors:

- Using a different Base URL than the one configured.
- Mismatching the leading `/` in a relative path.

## 🚀 **Module key takeaways**

- **API** = Application Programming Interface; communication is **menu (docs) → order (call) → delivery (response)**.
- **REST vs SOAP** — REST uses JSON (typically), SOAP only uses XML and is heavier. Workato supports JSON, XML on REST, and URL-encoded forms.
- **Three aspects of HTTP communication**: Authorization, Request, Response.
- **Status codes** — 1xx info, 2xx success, 3xx redirect, **4xx client error, 5xx server error**.
- **Auth types** — 9 supported, including Basic, Header, Query params, OAuth 2 (two grants), and AWS variants.
- **Base URL = Base URL + Action Request URL = Final URL**. Configuring Base URL restricts outbound traffic, secures data, and simplifies API version changes. **Relative paths require a configured Base URL.**
