## 🔄 **New Event via Polling**

The HTTP Connector supports **New Event via Polling** — each new or updated record is processed as a trigger event.

> ⚠️ **As a best practice, Workato does NOT recommend using this feature.**

The reasons matter:

- The number of events retrieved per poll depends on the API endpoint receiving the request — you don't control batch size cleanly.
- **The trigger does not support pagination.** If the API enforces pagination, this trigger will only read the **first page** of records — silently missing the rest.

---

### 🧠 Quick recall

- Should you use HTTP Connector's New Event via Polling trigger for production recipes? (No — not recommended.)
- Why is the lack of pagination support dangerous? (If the API paginates, only the first page of records is read — the rest are silently missed.)

## 🔒 **Sensitive data**

Day-to-day work with HTTP connections involves sensitive data — usernames, passwords, tokens, client IDs. Handling it carelessly can expose it through the recipe itself.

> 📌 Sensitive data can be exposed in three ways through a Workato recipe:
> 
> - To other users within your **Workato Team environment**.
> - When a recipe is **shared in the Workato Community Library**.
> - When a recipe is **cloned and copied** into another Workato account — data in recipe steps gets copied too.

> ⚠️ **NEVER hard-code sensitive data into input fields.** This is the single most important rule.

---

### 🛡️ Recipe security best practices

Three practices that together mitigate exposure:

1. **🔑 Set up authentication with query parameters** — don't put credentials in recipe step input fields.
2. **💊 Use datapills to pass sensitive data** between systems instead of literal values.
3. **🏦 Store sensitive data only in connection or account properties** — Workato treats these as protected storage, separate from recipe step data.

The result when you follow all three:

- ✅ Sensitive data isn't exposed to other users in your Workato Team environment.
- ✅ Sensitive data isn't exposed when a recipe is cloned.
- ✅ If you publish the recipe to the Community Library, no sensitive data leaks.

Bonus: centralized storage of sensitive info is also much easier to maintain and rotate.

---

### 🧠 Quick recall

- Name the three ways sensitive data can leak through a Workato recipe. (Other users in your Team environment; sharing to the Community Library; cloning/copying recipes)
- The hard rule about hard-coding sensitive data is: `_____`. (Never do it)
- Where should sensitive data actually be stored? (In connection or account properties — not in recipe step input fields.)
- Name the three security best practices. (Auth via query params; pass sensitive data via datapills; store sensitive data only in connection/account properties)

---

## 🚀 **Module key takeaways**

- **Avoid New Event via Polling** as an HTTP Connector trigger — it doesn't paginate and silently misses records past the first page.
- **Sensitive data in recipes is a real exposure risk** — three vectors: team members, Community Library publishing, and recipe cloning.
- The non-negotiable rule: **never hard-code sensitive data in input fields**.
- **Three best practices** to follow together: query-param auth, datapills for sensitive transport, and centralized storage in connection/account properties.
