## 🔄 **RLCM stages overview**

The **Recipe Lifecycle Management (RLCM)** process defines the steps to deliver automations from idea to production — improving team effectiveness while reducing business risk.

> 📌 The lifecycle has four stages:
> 
> 1. **📥 Backlog** — new automation projects and change requests are assessed and prioritized.
> 2. **🛠️ Development** — dev team implements and runs unit tests in the **dev environment**.
> 3. **🧪 Test** — QA team or business users perform end-to-end validation in the **test environment**.
> 4. **🚀 Production** — passing changes are scheduled, approved, and deployed to **prod**.

Changes that fail at the test stage return to the development queue for further modifications.

> ⚠️ Beyond separate Workato environments, you also need **separate dev/test/prod data sets** for the connected applications — so the dev team can't read or modify production systems, and untested changes can't break live automations.

> 📌 The three lifecycle phases in this lesson — **Discover**, **Implement**, **Operate** — map roughly to backlog→dev, dev→test, and prod-ongoing respectively.

---

## 🔍 **Discover**

Pool ideas from different teams and create **central visibility** of all active automation projects across the organization. This avoids overlapping projects, reduces duplicate work, and drives reuse.

Track automation change requests as **independent projects**. Tools that work:

- Project management platforms — **Jira**, others.
- Simple alternatives — shared spreadsheets, kanban boards.

> ✅ As your automation backlog grows, use Workato's **continuous value reporting methodology** to assess and prioritize projects.

> ✅ Build at minimum **two checkpoints** into the workflow:
> 
> - **Recipe review** before promoting dev → test.
> - **Release approval** before deploying → production.

---

### 🕸️ Managing dependencies with the dependency graph

> 📌 Use Workato's **dependency graph** to quickly identify and assess the impact on shared assets (callable recipes, connections, lookup tables) before making changes.

Even minor changes in a shared environment can cause unexpected production issues. The dependency graph surfaces what touches what — essential for safely changing shared assets.

## 🛠️ **Implement**

Two practices anchor the implementation phase: **recipe review** and **testing**.

---

### 📋 Recipe review

A **recipe review** is an essential checkpoint that ensures standards, security guidelines, and quality. Example questions a reviewer should answer:

- Do connections use **approved service accounts**, not individual team member accounts?
- Are appropriate teams **notified on unrecoverable errors** using the standard notification mechanism?
- Are naming conventions followed?
- Are error handling and design best practices in place?

> ✅ Use Workato's **recipe starter checklist** as a baseline; customize it over time.

> ✅ Optionally, automate reviews using an **external Git-based version control system** (e.g. via pull requests) as part of your CI/CD pipeline.

Project leads can request fixes in dev, or approve promotion to test.

---

### 🧪 Testing best practices

> ⚠️ Inadequate testing causes productivity loss, system outages, data inconsistencies, and potentially compliance violations (laws, internal policies, external regulatory requirements).

Recipe testing best practices:

- **🗺️ Understand the end-to-end workflow** and applications involved to identify test scenarios.
- **🛤️ Cover each path** in the workflow based on different business rules.
- **❌ Include negative test scenarios** for exception handling validation.
- **📋 Each test case** should include test data, preconditions, and expected outcomes.
- **🔒 Plan for non-functional testing** — e.g. security testing for API endpoints.
- **🔁 Test every time there's a change** to a recipe.

---

### 🤖 Recipes testing recipes

> 📌 You can use **Workato recipes to test other Workato recipes**. Regression-test-worthy scenarios can be fully automated.

Pattern:

1. Put business logic in a **callable recipe** (e.g. "create lead in CRM").
2. Build a separate **test recipe** that calls the business logic recipe with **mock data**.
3. Test recipe **asserts results** against expected outcome (e.g. lead created with mock fields).
4. Group related test recipes into a **test suite**.
5. Launch test suites via **webhook from CI/CD** or on a schedule.

Pay special attention to **test data** when automating tests, to ensure coverage.

## 🚀 **Operate**

The goal of the Operate phase: **enhance production stability and reduce post-implementation incidents** — by automating the response to issues, not handling them manually.

---

### 🤖 RecipeOps for automated monitoring and recovery

> 📌 **RecipeOps by Workato** automates the monitoring and recovery of mission-critical recipes.

Pattern:

1. Maintain a list of important recipes in a **lookup table**.
2. Build an automation using the **RecipeOps connector** to monitor those recipes' status.
3. On critical errors, RecipeOps can **restart the recipe** or **notify the operations team** for intervention.

You can also use RecipeOps to manage **third-party application planned downtime**:

- Maintain the downtime schedule in a lookup table.
- RecipeOps stops and starts recipes automatically before and after the maintenance window.
- On restart, Workato resumes processing unprocessed business events from where it left off.

---

### 📝 Compliance audits with activity audit logs

Regular **compliance audits** of production should be part of recipe lifecycle processes — ensures regulatory adherence and good governance.

> 📌 The Workato **activity audit log** is visible to account owners and users with **Admin role**. It is available as an **add-on**.

You can review changes made to accounts, connections, folders, recipes, packages, and more. Common audit use case: detecting accidental direct changes in production, then using the data to fix access policies.

---

### 📡 Streaming logs to external SIEM tools

If your teams use centralized operations or analytics tools (**Splunk, Datadog**, etc.), you can set up **real-time streaming** of recipe job data + user activity into them.

> ✅ Enrich the streams by including logs from related business systems — gives a **holistic view of business-process workflows** across multiple systems, so operations can troubleshoot without logging into each one.

You can also **customize the audit log message** before streaming — add event metadata (source name, environment identifier like `staging`/`prod`) to drive automated actions in your SIEM (e.g. alert different teams based on the environment).

## 🚀 **Module key takeaways**

- **4 RLCM stages**: backlog → dev → test → prod. **Two critical checkpoints**: recipe review (dev→test), release approval (→prod).
- **Discover** with the dependency graph — surface what touches what before changing shared assets.
- **Implement** with recipe reviews + thorough testing. Use **callable recipes as business logic, test recipes as harnesses** — recipes can test other recipes.
- **Operate** with RecipeOps (automated monitoring/recovery + planned-downtime handling) and activity audit logs (compliance + change tracking, Admin-only add-on).
- **External SIEM integration** (Splunk, Datadog) — stream Workato logs with custom metadata for cross-system observability.
