## 🚢 **Release management overview**

A thorough release management process ensures changes land smoothly in production. It answers three key questions:

- **🏷️ How to identify** different releases.
- **📦 What should be the unit of deployment.**
- **📢 How to manage and communicate release policies** across the organization.

As the number of recipes and shared assets grows, the number of packages promoted between environments grows too — release policies become essential for **traceability** and **safe production deployments**.

---

## 🏷️ **Release types**

> 📌 A **release** is a set of changes grouped together for deployment to production. Every release classifies as one of **three types**:

|Type|What it includes|Testing required|
|---|---|---|
|**🚀 Major**|New automation or major enhancements; all assets built, tested, and implemented together|**Regression-tested** to ensure zero impact on existing behavior|
|**🔧 Minor**|Only changed assets since the last release|**Limited testing**; minimal impact on existing business processes|
|**🩹 Bug-Fix**|Production issue fixes — configuration changes OR recipe changes|Focused on the specific issue|

---

## 🔢 **Release identification**

Once your release policy is defined, identify each release uniquely using a **semantic versioning policy**. Include a reference to the functional area or business process, plus a version number with three parts based on release type.

> 📌 **Versioning format**: `v.MAJOR.MINOR.PATCH`
> 
> _Example:_ `Sales-Lead-Mgmt-v.1.0.0`

Unique identification helps schedule parallel releases by managing shared dependencies and communication between project teams.

---

### 🗄️ Where to store release history

> 📌 **Two ways to track release details:**
> 
> - **Git tags** — automated release-history management in a Git-based version control system. Tags are like bookmarks pointing to specific release versions.
> - **Shared drive copies** — if not using external version control, rename exported packages per your release identification policy and store them in **Box, a network drive**, or similar.

## 📦 **Manifest as the unit of deployment**

> 📌 **Folders are the unit Workato uses for import/export between environments.** When creating a manifest, you select the Home folder or any folder under it.

**Best practice:** create manifests at **top-level folders** used for functional grouping of project-specific automations. This assumes shared assets (callable recipes, connections) are maintained in a top-level folder.

Benefits of this approach:

- **🗂️ Controlled count** — keeps the number of manifests in a given environment manageable.
- **♻️ Manifest reuse** — the same manifest can be reused for subsequent releases to add or update assets.

Alternative: for a project folder with no external dependencies, create a manifest from the project-specific folder.

> ⚠️ **Avoid renaming or moving assets between releases** whenever possible — it can break live automations.

## 📋 **Release planning — the five plans**

A well-defined process ensures release policies are communicated and enforced consistently. Five plans cover the full release lifecycle:

|Plan|Covers|
|---|---|
|**🛠️ Build & Test Plan**|Approves changes, test results, user acceptance criteria|
|**🚀 Deployment Plan**|Release package, rollback plan, production freeze schedule|
|**✅ Validation Plan**|Deployment approval, release success validation|
|**📢 Communication Plan**|End-user communication, end-user training|
|**🔄 Review Plan**|Monitor release, improve metrics, lessons learned, feedback for next release|

> 📌 All five plans are required — name them as a mnemonic: **Build → Deploy → Validate → Communicate → Review**.

---

## 📊 **Measure and improve**

Once you've defined the recipe lifecycle process, **measure its effectiveness** and identify improvement areas. Capturing key milestones across projects reveals process gaps that you can address incrementally.

Useful metrics to track:

- **📈 Release approval ratio** — (approved releases) ÷ (total releases requested). A high rejection rate means you need to fine-tune recipe review or testing stages.
- **🏢 Rejected releases by functional group** — identifies training needs for specific teams.
- **🚨 Incidents related to a given release** — improves quality control checks.

---

## ⚙️ **CI/CD automation**

Once the RLCM process is well-defined, automate it with **DevOps best practices**.

> 📌 **CI/CD (Continuous Integration / Continuous Delivery)** automates how teams build, test, and package automations — speeding up release cycles and improving quality.

**What Workato provides:**

- A rich set of **platform APIs** to fit any automation workflow.
- Integration with broad set of **DevOps platforms**: Jenkins, CodeShip, Bamboo, GitHub, and others.
- Workato recipes can orchestrate the CI/CD pipeline directly, OR be used alongside other CI/CD tools.

---

### ✅ CI/CD design best practices

Regardless of the implementation tools chosen, three things should be automated:

- **🤖 Automate package builds** — users with access can easily trigger the CI/CD pipeline.
- **🤖 Automate quality control** — regression testing, load testing; teams immediately know if a build passed or failed.
- **🤖 Automate deployment** — across all environments (dev → test → prod).

## 🚀 **Module key takeaways**

- **Three release types**: Major (regression-tested, zero-impact), Minor (changed assets only, limited testing), Bug-Fix (production issue fixes).
- **Versioning**: `v.MAJOR.MINOR.PATCH` (semantic), e.g. `Sales-Lead-Mgmt-v.1.0.0`. Track history with Git tags or shared drive copies.
- **Manifests = unit of deployment**, created at top-level folders. Don't rename/move assets between releases.
- **Five release plans**: Build & Test, Deployment, Validation, Communication, Review.
- **Measure** with release approval ratio + per-team rejection counts + incidents-per-release.
- **CI/CD** automates the lifecycle — Workato APIs integrate with Jenkins, GitHub, Bamboo, CodeShip, etc.
