## 📦 **Manifests and packages**

> 📌 A **manifest** is a **record of all the selected recipes and their dependencies** — application connections, lookup tables, account properties, and other assets.

> 📌 A **package** is the **exported zip file** built from a manifest. It includes all the selected recipes and their dependencies, ready to be deployed to another environment.

**Workflow:** create a manifest → export a package → import that package into the target environment to promote your changes.

> ⚠️ **Exported packages do NOT include credentials** — for security reasons. After importing into a target environment, connections must be re-authenticated there.

---

## 📋 **What a package can include**

A package may contain any of:

- **🍳 Recipes**, including **callable recipes**
- **🔌 Application connections**
- **🔎 Lookup tables**
- **📨 Message templates**
- **📐 Common data models**
- **📡 Pub/sub topics**
- **🛠️ SDK connectors**
- **🔑 Account properties**
- **🌐 API collections**

Once you're happy with the dev-side activities for a given automation request, it's time to **promote the change to the test environment** — and the manifest/package pair is how you do it.

## 🚀 **Module key takeaways**

- **Manifest** = the list (what to include).
- **Package** = the exported zip (the deliverable).
- **Manifest → Package → Import into target environment** is the deployment pattern between environments.
- **No credentials in exports** — connections must be re-authenticated in the target.
- A package can include 9 different asset types — far more than just recipes.
