## 🌊 **The journey toward data orchestration**

In the 1980s, computer scientists began designing systems for operability between patchwork databases — what we'd later call **digital transformation**, using digital technologies to drive business efficiency. Those early days were labor-intensive: manual data entry, batch operations, ad hoc workflows across disparate systems. The goal was always the same — efficient business processes.

Today the landscape includes pipeline patterns, streaming data, APIs, cloud integration for flexibility and scalability, and AI/ML. **Data orchestration** is the modern evolution: it sits on top of traditional integration and automation, adding unified visibility, governance, and the ability to move data and insights bidirectionally across modern data architectures.

---

## 🗄️ **Workato as a data platform**

Most organizations now adopt **data platform** tools to move data from disparate sources into a centralized **data warehouse, data lake, or lakehouse**. From there, data becomes available to consumers through dashboards, business applications, APIs, LLMs, and more.

Around the data platform, organizations also use:

- **🎼 Orchestration tools** — manage the flow of data across systems to ensure efficiency.
- **🛡️ Data governance tools** — build and enforce policies, ensure data quality.
- **👀 Observability tools** — monitor health and performance.

> 📌 Workato is uniquely positioned because it does the full loop: **ingest and transform** data from sources into the warehouse/lake/lakehouse, **and push insights back out** into business applications or processes for end-user or app consumption — with end-to-end orchestration and visibility at each step.

## 🔗 **Tools for connectivity**

Workato has dozens of connectors, built-in tools, and community recipes built specifically for data orchestration. The connectivity tools below are the ones most commonly used in a data pipeline.

---

### 📦 Batch processing

Batch processing efficiently moves large volumes of data across systems, significantly increasing the speed of loading and operations.

> 📌 A batch action is a **synchronous** action that processes a limited number of records at a time. When designing recipes, **always match a batch trigger with a batch action** — mismatching them breaks the batching.

> ⚠️ Only **certain Workato connectors** support batch processes out-of-the-box — not all of them.

_Example:_ retrieving many Salesforce records for use in other systems. Instead of retrieving one record at a time, you obtain many records in batches.

---

### 📁 Workato FileStorage

**Workato FileStorage** is a built-in utility for storing and using files within Workato, built on an encrypted file storage system.

> 📌 **Hard limits:** maximum file size **10 GB**, maximum workspace storage repository **100 GB**.

Common uses:

- Store files from any connector.
- Receive data as a **bulk CSV stream** from Salesforce actions and store it as a CSV file.
- Persist and fetch static files that are frequently referenced in recipes.
- Perform CSV transformations by pairing FileStorage with the **SQL Transformation** built-in utility.

_Example:_ continuing the Salesforce batch case above, you could store the retrieved records temporarily in CSV or XML, making them accessible to Workato without needing a database.

---

### 📊 Workato Data Tables

**Workato Data Tables** let you store frequently used and frequently referenced data efficiently.

✅ **Good for:** persisting data, faster responses, Workflow Apps data, reference tables that need to be accessible from within Workato.

❌ **Not recommended for:** relational database solutions, large data sets, or sensitive data.

---

### 🔒 OPA smart shunt

**OPA (On-Premise Agent) smart shunt** sends data directly from a source system to a destination that is **entirely on-premise**.

> 📌 This is **"ground-to-ground" integration** — your secure data **never leaves the data center**. This keeps you compliant with strict security requirements while still using Workato to orchestrate the movement.

Optimized for **high volume and high speed** data transfer, and also useful for keeping operations across multiple on-premise databases organized.

---

### 🚀 Enterprise Data Hub Accelerator

A collection of prebuilt assets to fast-track centralized data hubs:

- **Prebuilt integrations** with Snowflake and a variety of data sources to rapidly centralize, enrich, and cleanse data.
- **Data validation rules & triggers** — configuration-based validation for all ingested records, with triggers to notify stakeholders of issues.
- **Customizable data models** — usable out-of-the-box but tailorable for Customer 360 or Employee Data Hub projects.

---

### 🧠 Quick recall

- Workato FileStorage max file size is `_____` GB; max workspace storage is `_____` GB. (10 GB; 100 GB)
- A batch trigger must be paired with a `_____`. (Batch action — never mix batch trigger with single action or vice versa)
- Sensitive on-premise data must stay inside the data center. Which Workato tool fits? (OPA smart shunt — ground-to-ground integration)
- You need a reference table accessible from inside recipes. Which tool? (Workato Data Tables)
- Should Workato Data Tables hold a large relational dataset? (No — not recommended for large data sets, relational use cases, or sensitive data.)

## 🔄 **Tools for data transformation**

### 🧮 SQL Transformations

**SQL Transformations** apply transformations to **large volumes** of data using SQL queries.

- Query data coming in from any number of data sources.
- **No volume limit** — no cap on how much data you fetch, transform, or output.
- High-performance execution of queries and output generation.
- Supports complex `SELECT` queries with joins and other SQL functions.

---

### 📋 SQL Collection

**SQL Collection** is a native Workato application providing the tools to manipulate table data — useful for synchronizing related data across multiple systems (databases, web services, etc.).

It functions as a **temporary SQL database**:

- Supports statements in SQL syntax.
- Allows SQL-based transformations for merging data from different sources.

> ⚠️ **SQL Collection lists and tables are NOT persistent — data is deleted after the job run.** Use it as a scratch space, not a store.

> 📌 **Soft limit: 50,000 records** per job.

---

### ⚖️ When to use which

| | 📋 **SQL Collection** | 🧮 **SQL Transformations** |
|---|---|---|
| **Best for** | In-memory SQL on small-to-medium datasets | High-performance processing on very large datasets |
| **Data size** | Up to ~50,000 rows | Millions of records |
| **Persistence** | Temporary — deleted after job run | Pairs with FileStorage for persistence |
| **Typical use** | Quick aggregations, temporary data manipulation within a recipe | Complex SQL logic, Change Data Capture (CDC), heavy workloads |

> 📌 **Rule of thumb:** if the dataset fits under ~50K rows and you only need it for the current job → SQL Collection. If it's larger, needs to persist, or requires CDC-style processing → SQL Transformations (typically paired with FileStorage).

---

### 🧠 Quick recall

- SQL Collection's soft limit is `_____` records. (50,000)
- Where does SQL Collection data go after the job ends? (It's deleted — SQL Collection is not persistent.)
- Millions of records need to be transformed with complex joins and persisted for downstream processing. Which tool? (SQL Transformations, typically paired with FileStorage.)
- What's CDC, and which tool fits it? (Change Data Capture — capturing record-level changes for downstream sync; use SQL Transformations.)

## 🚀 **Module key takeaways**

- **Workato as a data platform** does the full loop: **ingest into** the warehouse/lake/lakehouse, then **push insights back out** to business apps.
- **Connectivity tools** — batch processing (sync batch trigger ↔ batch action), FileStorage (10 GB file / 100 GB workspace), Data Tables (small reference data only), OPA smart shunt (ground-to-ground, never leaves the data center), Enterprise Data Hub Accelerator (prebuilt centralization assets).
- **Transformation tools** — SQL Transformations for large datasets and CDC; SQL Collection as a temporary in-memory SQL workspace (~50K row soft limit, deleted after the job).
- **Decision rule** — small + temporary → SQL Collection; large + persistent → SQL Transformations + FileStorage.
